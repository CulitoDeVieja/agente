Slide 2 · Visión

El objetivo real

Construir la app es la excusa. El experimento: medir si 4 agentes Claude en loop pueden planificar, curar skills, programar y auditar un proyecto real sin chocarse.

Lo que probamos

Coordinación distribuida

4 agentes en 4 máquinas, repo git como única memoria compartida, tareas markdown como cola. Sin orquestador central vivo: el repo ES el orquestador.
Lo que aprendemos

Efectividad del patrón

Qué tan autónomo es el sistema, cuánto overhead tiene, qué tipo de tarea se adapta mejor, dónde se atascan los agentes.
Slide 3 · Roles

4 agentes en loop · 2 on-demand

#0 Orchestrator
local
Consolida, unifica, detecta errores, escala al owner. Único que responde directo.
#1 Skills-Curator
VPS-1 · Hostinger
Mantiene biblioteca de skills técnicas. Busca, crea, mejora patrones reusables.
#2 Builder
VPS-2 · Hostinger
Programa. Implementa specs. Escribe tests. Codea el árbol real del proyecto.
#3 Auditor + Ops
VPS-3 · Hostinger
QA estática + dinámica. Deploy. Release. Soporte post-lanzamiento.

On-demand: Architect (spec + ADR) · Scout (research de mercado/usuarios).

Slide 5 · Tech

Stack elegido (por orchestrator)

Frontend

React 18 + TS + Tailwind v4

Vite como bundler. Contexto para estado global. Sin state manager externo (ADR-001).
Backend embebido

Rust (Tauri 2)

8 comandos IPC: list_tasks, read_state, read_task, git_pull, git_log, git_status, get_config, set_config.
Build + Release

.msi + .exe Windows

Target <15MB. Nativo, sin dependencia Node en runtime. Futuro: PWA web.
